Interview Prep — CORS Origin Reflection

Common interview questions, model answers, one-liner bypasses, and red flags for CORS misconfiguration reviews.

30-Second Explanation

CORS lets servers grant cross-origin JavaScript read access to their responses. The server signals permission with the Access-Control-Allow-Origin (ACAO) header. A vulnerable pattern is origin reflection: the server copies the incoming Origin request header directly into the ACAO response header. Combined with Access-Control-Allow-Credentials: true, this lets any attacker-controlled page issue authenticated fetch requests and read the response — bypassing the entire purpose of the Same-Origin Policy. The fix is an explicit allowlist of trusted origins validated with exact string equality.

Q & A

What is CORS, and why does the browser enforce it? ▼
CORS (Cross-Origin Resource Sharing) is an HTTP-header mechanism that lets servers relax the Same-Origin Policy for specific trusted origins. Without CORS headers, browser JS can initiate a cross-origin request but cannot read the response — the browser enforces this restriction to prevent malicious pages from silently reading authenticated data on other sites the user is logged into. Servers opt in to cross-origin reads by emitting Access-Control-Allow-Origin in their response.
What makes "origin reflection" dangerous? ▼
Origin reflection means the server copies the Origin request header verbatim into Access-Control-Allow-Origin. When paired with Access-Control-Allow-Credentials: true, this effectively tells the browser: "trust any origin and allow it to read credentialed responses." Because the browser includes session cookies on the request, the attacker's page can read the victim's authenticated data — bypassing CORS entirely.
Why can't you use Access-Control-Allow-Origin: * with credentials? ▼
The Fetch specification explicitly forbids it. When Access-Control-Allow-Credentials: true is set, the browser requires ACAO to be an exact, non-wildcard origin. The wildcard (*) means "allow any origin without credentials." Allowing both credentials and a wildcard origin would be equivalent to removing the SOP entirely. Developers who need to support many origins and credentials must implement a server-side allowlist and reflect the matched origin exactly.
How would you find a CORS reflection vulnerability in a bug bounty target? ▼
  1. In Burp Suite, browse the authenticated app and look for API responses with CORS headers.
  2. In the Repeater, add a synthetic Origin: https://evil.com header to any authenticated request.
  3. Check whether the response echoes it back in ACAO and also has ACAC: true.
  4. Confirm exploitability: write a PoC page on your server that fetches the endpoint with {credentials:'include'} and verifies you can read the authenticated response body.
How do you fix origin reflection? ▼
Maintain a server-side allowlist of trusted origins (exact strings) and compare the incoming Origin header against it with strict equality. Only set ACAO to the matched origin — never to a reflected value or wildcard when credentials are involved. Reject all other origins by omitting the CORS headers entirely. 
const ALLOWED = ['https://app.example.com', 'https://admin.example.com'];
if (ALLOWED.includes(req.headers.origin)) {
  res.setHeader('Access-Control-Allow-Origin', req.headers.origin);
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  res.setHeader('Vary', 'Origin');  // cache correctness
}

Developer Fix

Exact allowlist with Vary header

The Vary: Origin header is required so CDNs and proxies don't cache a response with one origin's ACAO and serve it to a different origin. 

const ALLOWED_ORIGINS = new Set([
  'https://app.example.com',
  'https://admin.example.com',
]);

app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (ALLOWED_ORIGINS.has(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Access-Control-Allow-Credentials', 'true');
    res.setHeader('Vary', 'Origin');
  }
  next();
});

Red Flags in Code Review

⚠ res.setHeader('Access-Control-Allow-Origin', req.headers.origin) — unconditional reflection
⚠ Access-Control-Allow-Origin: * combined with Access-Control-Allow-Credentials: true
⚠ CORS handler that sets ACAO to the origin from a regex match or substring check, not exact equality
⚠ Missing Vary: Origin header when origin-dependent CORS headers are set
⚠ Internal/admin APIs with ACAC: true accessible from any origin
