CORS Origin Reflection

The victim API blindly reflects whichever Origin header the browser sends back as Access-Control-Allow-Origin, combined with Access-Control-Allow-Credentials: true. Any origin can read the authenticated response — including one you control.

🔎 What to Look For

After clicking "Fetch Account", open the Network tab in DevTools and inspect the response headers. The dangerous pattern is when both of these appear together:

Access-Control-Allow-Origin: https://attacker.example.com ← reflects your origin verbatim Access-Control-Allow-Credentials: true ← allows cookies to be read

A browser will only expose a credentialed response to JavaScript when ACAO is an exact non-wildcard origin and ACAC is true. Reflecting the caller's origin satisfies both conditions for any caller.

Why wildcard + credentials is impossible: The spec forbids Access-Control-Allow-Origin: * together with credentials. Reflection is a misguided workaround that destroys the protection.

📜 Vulnerable Server Code 
// routes/x1.js — the vulnerable CORS handler
app.get('/api/x1/account', (req, res) => {
  // BUG: reflect every Origin header back as ACAO
  const origin = req.headers['origin'];
  if (origin) {
    res.setHeader('Access-Control-Allow-Origin', origin); // ← any origin allowed
    res.setHeader('Access-Control-Allow-Credentials', 'true');
  }

  // Return authenticated data
  const user = req.session?.user || { username: 'victim', role: 'user' };
  res.json({
    username: user.username,
    email:    'victim@example.com',
    role:     user.role,
    flag:     'HUNT{origin_reflection_leaks_creds}',
  });
});
