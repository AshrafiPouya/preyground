Interview Prep — postMessage Security

Interview questions on the postMessage API security model, event.origin checks, and DOM-XSS via postMessage sinks.

30-Second Explanation

window.postMessage() is a deliberate exception to the Same-Origin Policy: any window can message any other window regardless of origin. The browser provides event.origin so the receiver can verify who sent the message — but this check must be implemented by the developer. Omitting it means any page that gets a reference to your window (via window.open(), window.opener, or iframes) can exfiltrate data or inject HTML by sending crafted messages. The fix is always to check event.origin at the top of every message listener, and to use a specific targetOrigin (never '*') when sending sensitive replies.

Q & A

What is postMessage and why is it an exception to SOP? ▼
window.postMessage(data, targetOrigin) allows a script to send a message to another window (e.g., a popup or iframe) regardless of whether they share an origin. It was introduced to allow legitimate cross-origin communication — for example, a widget embedded in an iframe communicating with its parent page. Unlike cross-origin fetch, postMessage is not blocked by SOP at the sending side; security is entirely opt-in at the receiving side via event.origin validation.
What is event.origin and why must you always check it? ▼
event.origin is the origin of the window that called postMessage — set by the browser and cannot be spoofed by the sender. Without checking it, your listener trusts messages from every origin on the internet. An attacker who can get a reference to your window (through a link, popup, iframe, or window.opener) can send arbitrary data and trigger any branch of your listener — including data exfiltration or DOM-XSS. The check must be the very first thing in the listener: 
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://trusted.example.com') return;
  // safe to process event.data
});
What is the difference between event.origin and targetOrigin? ▼
They operate in opposite directions:
  • event.origin — on the receiving side: who sent this message? Use it to reject untrusted senders.
  • targetOrigin (second arg to postMessage) — on the sending side: only deliver this message if the recipient window is at this origin. Use '*' only for non-sensitive data; use a specific origin for sensitive replies.
Using targetOrigin: '*' when sending a reply containing a secret is dangerous: any window that races to acquire the reference could be at any origin and will receive the secret.
How does a postMessage handler become a DOM-XSS sink? ▼
When a listener passes event.data (or a property of it) into a sink like innerHTML, document.write, eval, or a URL-based attribute (href, src), an attacker who can send a message can inject arbitrary HTML/JS. This is called a postMessage-based DOM-XSS vulnerability. Burp Suite's DOM Invader specifically probes for these by injecting canary strings via postMessage and watching for DOM mutations. The fix: sanitize or avoid HTML sinks entirely; use textContent instead of innerHTML for untrusted data.
How would you find postMessage vulnerabilities in the wild? ▼
  1. Search source for addEventListener('message' and check if each listener validates event.origin.
  2. Use Burp DOM Invader: enable postMessage interception and inject canary strings.
  3. Audit sinks: does the listener pass event.data to innerHTML, eval, or URL-setting APIs?
  4. Check senders: search for postMessage( with targetOrigin: '*' in replies to sensitive messages.
  5. Check window.open() usage — if the opened page is cross-origin and replies with secrets, can you intercept by racing with a same-name window?

Bypass One-Liners

const w = window.open('http://victim.com/widget.html'); setTimeout(() => w.postMessage(JSON.stringify({type:'getSecret'}), '*'), 800); window.addEventListener('message', e => console.log(e.data));

Opens victim page and exfiltrates secrets if listener has no origin check.

// DOM-XSS via postMessage: window.open('http://victim.com/page.html'); setTimeout(() => w.postMessage(JSON.stringify({type:'render', html:'<img src=x onerror=alert(1)>'}), '*'), 800);

Developer Fix

Always validate event.origin — both directions

// Receiving side — check origin first
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://trusted.example.com') return; // reject

  let msg;
  try { msg = JSON.parse(event.data); } catch { return; }

  if (msg.type === 'getProfile') {
    // Reply only to the trusted origin
    event.source.postMessage(
      JSON.stringify({ user: 'alice' }),
      event.origin   // specific targetOrigin, not '*'
    );
  }

  if (msg.type === 'render') {
    // Use textContent, never innerHTML, for untrusted data
    document.getElementById('output').textContent = msg.text;
  }
});

Red Flags in Code Review

⚠ addEventListener('message', handler) with no event.origin check inside the handler
⚠ event.source.postMessage(sensitiveData, '*') — reply sent to any origin
⚠ element.innerHTML = event.data or any property of event.data — DOM-XSS sink
⚠ eval(event.data), document.write(event.data), or setting location from message data
⚠ Partial origin check: if (event.origin.includes('trusted')) — substring bypass
