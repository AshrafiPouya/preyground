Interview Prep — Reverse Tabnabbing

Interview questions on window.opener, reverse tabnabbing attacks, rel=noopener, and related phishing vectors.

30-Second Explanation

When a page opens a new tab with target="_blank", the opened page receives a reference to the original tab via window.opener. Cross-origin, the opened page cannot read the opener's DOM — but it can set window.opener.location to redirect the original tab to any URL. This is reverse tabnabbing: the victim clicks a link, switches back to the original tab, and finds it now showing a phishing page. The fix is trivially rel="noopener noreferrer" on every target="_blank" link, which nulls out the window.opener reference in the new tab.

Q & A

What is window.opener? ▼
window.opener is a reference to the window object that opened the current window — either via window.open() or a link with target="_blank". When both windows share the same origin, the opener reference grants full DOM access. Cross-origin, the Same-Origin Policy restricts most properties — but the location property is writeable even cross-origin, allowing the opened page to redirect the opener tab.
Explain the reverse tabnabbing attack step by step. ▼
  1. Victim browses to a trusted site that renders user-submitted links with target="_blank".
  2. An attacker submits a link pointing to their malicious page.
  3. Victim clicks the link — attacker page opens in a new tab. Victim switches to the new tab.
  4. Attacker page JS executes: if (window.opener) window.opener.location = 'https://phishing.example.com'
  5. Original tab silently redirects to the phishing clone while the victim is looking at the new tab.
  6. Victim switches back — sees the phishing login page, believes they were logged out, enters credentials.
The attack works because redirecting opener.location is allowed cross-origin by spec.
What does rel="noopener" do, and why should you also add noreferrer? ▼
  • noopener: Instructs the browser to set window.opener = null in the opened page, removing the reference to the originating tab. This is the direct fix for tabnabbing.
  • noreferrer: Prevents the browser from sending the Referer header, hiding the originating URL from the destination server. Older browsers that don't support noopener will also omit the opener when noreferrer is set.
Best practice: always use rel="noopener noreferrer" together to cover both concerns and legacy browsers.
Do modern browsers protect against this automatically? ▼
Partially. Modern Chromium-based browsers (Chrome 88+) and Firefox implicitly apply rel=noopener to cross-origin target="_blank" links even without the attribute, following the HTML spec update. However:
  • Older browsers and Safari have inconsistent behaviour.
  • Same-origin links are not implicitly protected.
  • Relying on browser defaults is fragile — explicit rel attributes are canonical and correct.
Always add rel="noopener noreferrer" explicitly; don't rely on browser version.
How would you scan a large codebase for missing noopener? ▼ 
# Find target="_blank" links missing rel="noopener"
grep -rn 'target="_blank"' . | grep -v 'noopener'

# Or with ripgrep (faster):
rg 'target="_blank"' --type html | grep -v noopener
Also check JS that constructs anchor elements: a.target = '_blank' without subsequently setting a.rel = 'noopener noreferrer'. Linting tools like eslint-plugin-react have rules for this (react/jsx-no-target-blank). CSP does not mitigate this — it's a structural fix only.

Attack One-Liners

// attacker page opened via target="_blank" link with no rel="noopener": if (window.opener) window.opener.location = 'https://phishing.attacker.com/login';

Runs on page load — silently redirects the victim's original tab.

// More stealthy — delay to give victim time to switch tabs: setTimeout(() => { if (window.opener) window.opener.location = 'https://phishing.example.com'; }, 2000);

Developer Fix

Always add rel="noopener noreferrer" to target="_blank" links

<!-- Vulnerable -->
<a href="{{ userUrl }}" target="_blank">Visit</a>

<!-- Fixed -->
<a href="{{ userUrl }}" target="_blank" rel="noopener noreferrer">Visit</a>

<!-- Dynamically created links in JS -->
const a = document.createElement('a');
a.href   = url;
a.target = '_blank';
a.rel    = 'noopener noreferrer';   // required
document.body.appendChild(a);
a.click();

Red Flags in Code Review

⚠ target="_blank" on any link without rel="noopener"
⚠ window.open(url) in JS without storing the return value and immediately clearing opener: child.opener = null
⚠ User-controlled URLs rendered as clickable links with target="_blank" (profile pages, comments, link-sharing features)
⚠ Template engines that auto-linkify URLs without appending rel="noopener noreferrer"
⚠ Relying on browser auto-noopener behaviour — implicit, not guaranteed across all browsers and same-origin links
