Interview Prep — Cross-Site WebSocket Hijacking

Senior-level interview questions on CSWSH, how it differs from CSRF, why SOP doesn't protect WebSockets, and correct server-side defenses.

30-Second Explanation

Cross-Site WebSocket Hijacking (CSWSH) exploits the fact that browsers include cookies in WebSocket upgrade requests but the Same-Origin Policy does not block cross-origin WebSocket connections. If the WebSocket server authenticates users via cookies and does not validate the Origin header on the upgrade request, any web page can silently open a WebSocket to the server as the authenticated victim and read all messages — including data the server believes it is sending only to trusted pages. This is analogous to CSRF for read-only channels, but more dangerous because the attacker can also exfiltrate the server's responses via ws.onmessage. The fix is server-side: validate the Origin header during the HTTP upgrade handshake, and additionally use CSRF tokens for WebSocket authentication if Origin alone is insufficient.

Q & A

How is CSWSH different from CSRF? ▼
Both attacks exploit the fact that browsers automatically include credentials (cookies) in cross-origin requests — but they target different channels:
Property CSRF CSWSH
Protocol HTTP WebSocket
Attacker can read response? No (SOP blocks) Yes (SOP doesn't apply to WS)
Primary goal State-changing actions Data exfiltration + actions
Common mitigation CSRF token / SameSite cookies Origin header validation
CSWSH is generally more severe: the attacker can both send messages as the victim and read the server's replies in real time.
Why doesn't the Same-Origin Policy protect WebSockets? ▼
The SOP was designed for the HTTP request-response model: the browser sends the request but blocks JS from reading a cross-origin response. WebSockets use a fundamentally different model — after the HTTP upgrade handshake, the connection becomes a persistent bidirectional stream, and all subsequent frames are outside the HTTP model the SOP was designed around.

Specifically: the browser does not block cross-origin WebSocket connection establishment. The Origin header is sent with the upgrade request, but it is up to the server to check and reject it. If the server doesn't validate the origin, the browser happily completes the handshake and gives the attacker full read/write access to the socket.
What headers should a WebSocket server validate on connection? ▼
  • Origin — the most important. Reject connections from unexpected origins with a 403 or a ws.close(4001, 'Forbidden').
  • Session / authentication token — cookies are sent automatically; but also support token-in-URL or a protocol-level handshake token as a secondary check.
  • CSRF token — for applications where Origin alone is insufficient (e.g., you need to trust multiple third-party origins), include a nonce in the WS URL or first message and validate it.
wss.on('connection', (ws, req) => {
  const origin = req.headers.origin;
  const ALLOWED = ['https://app.example.com'];
  if (!ALLOWED.includes(origin)) {
    ws.close(4001, 'Forbidden origin');
    return;
  }
  // … proceed
});
Does SameSite=Strict on the session cookie prevent CSWSH? ▼
SameSite=Strict or SameSite=Lax prevents the cookie from being sent on cross-site requests — which would block the attack if fully enforced. However, there are important nuances:
  • Browser SameSite support varies and has edge cases (e.g., top-level navigation for Lax).
  • The cookie may be served without a SameSite attribute — legacy behaviour defaults to None in older browsers.
  • SameSite is a defence-in-depth measure, not a replacement for Origin header validation on the server.
  • Best practice: use both SameSite=Strict cookies and server-side Origin validation.
How do you test for CSWSH in a bug bounty or pentest? ▼
  1. In Burp Suite, proxy the WebSocket upgrade request. In Burp Repeater, change the Origin header to an attacker domain. Observe whether the server accepts the connection.
  2. If accepted, write a minimal PoC HTML page that opens a WebSocket to the victim server and logs all onmessage data.
  3. Check for authentication: does the server send different data when the cookie is present vs. absent? This confirms the vulnerability is exploitable against an authenticated victim.
  4. Also test: send messages as the victim (state-changing operations over WS) to assess full impact.

Developer Fix

Server-side Origin validation on WebSocket upgrade

const ALLOWED_ORIGINS = new Set([
  'https://app.example.com',
  'https://admin.example.com',
]);

wss.on('connection', (ws, req) => {
  const origin = req.headers.origin;

  // Reject all unknown origins
  if (!origin || !ALLOWED_ORIGINS.has(origin)) {
    ws.close(4001, 'Connection refused: invalid origin');
    return;
  }

  // Optionally validate a CSRF token passed in the URL:
  const url   = new URL(req.url, 'http://localhost');
  const token = url.searchParams.get('token');
  if (!validateCsrfToken(req, token)) {
    ws.close(4002, 'Connection refused: invalid token');
    return;
  }

  // Authenticated, known-origin connection — proceed
  ws.send(JSON.stringify({ type: 'welcome' }));
});

Also set SameSite=Strict on session cookies as a defence-in-depth layer.

Red Flags in Code Review

⚠ wss.on('connection', (ws, req) => { ... }) with no req.headers.origin check
⚠ WebSocket authentication based only on cookies without any Origin or CSRF validation
⚠ Server sends sensitive per-user data in the onconnect welcome message to all connections
⚠ Session cookie without SameSite attribute — defaults to cross-site in many browsers
⚠ Assuming WebSocket is safe "because it's not HTTP" — the WS handshake is HTTP and shares HTTP cookie rules
