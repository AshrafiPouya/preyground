Interview Prep — CORS Null Origin & Weak Allowlist

Deep-dive questions on null origin sources, allowlist bypass techniques, and regex pitfalls in CORS configuration.

30-Second Explanation

Two classic CORS allowlist implementation bugs: (1) trusting the literal string "null" as a valid origin, and (2) using substring matching instead of exact equality. The null origin is sent by browsers for sandboxed iframes, data URIs, and certain redirects — any attacker page can force it. Substring matching (origin.includes('trusted.com')) can be bypassed by registering a domain like eviltrsted.com.attacker.io or prepending the trusted string. The correct approach is an explicit array of exact origins compared with strict equality.

Q & A

When does a browser send Origin: null? ▼
  • Sandboxed iframes (<iframe sandbox="allow-scripts">) — the most common exploitation vector
  • Data URIs (data:text/html,<script>...</script>)
  • Cross-origin redirects in some browsers
  • File protocol (file:// pages)
  • Opaque origins — browser's way of saying "origin intentionally withheld"
An attacker can reliably trigger Origin: null by wrapping their exploit in a sandboxed iframe with srcdoc.
How does substring matching break a CORS allowlist? ▼
If the check is origin.includes('trusted.example.com'), an attacker can bypass it with a hostname that contains the trusted string as a substring:
  • https://trusted.example.com.evil.io — trusted string is a subdomain prefix
  • https://evil.io/trusted.example.com — in a path (some naive parsers)
  • https://notrusted.example.com — contains the substring
Same risk applies to regex without anchors: /trusted\.example\.com/ matches evil-trusted.example.com. Always anchor: /^https:\/\/trusted\.example\.com$/.
Why is it wrong to trust null origin in production? ▼
The null origin represents an "opaque origin" — it does not identify a specific web page. Trusting it grants cross-origin read access to every sandboxed iframe on the internet. A legitimate use case for null in production is essentially non-existent: if you need cross-origin access, use a real, identifiable origin. Any code that sets ACAO: null should be treated as a critical vulnerability.
How do you safely implement a CORS allowlist for multiple origins? ▼ 
const ALLOWED = new Set([
  'https://app.example.com',
  'https://admin.example.com',
]);

// In middleware:
const origin = req.headers.origin;
if (origin && ALLOWED.has(origin)) {        // exact Set lookup
  res.setHeader('Access-Control-Allow-Origin', origin);
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  res.setHeader('Vary', 'Origin');
}
// If origin not in set: omit CORS headers entirely
Key points: use a Set for O(1) exact lookup, never use includes() or unanchored regex, always add Vary: Origin, and never fall back to reflecting an unknown origin.
How would you test for a null origin bypass on a bug bounty target? ▼
In Burp Suite Repeater, manually set Origin: null on an authenticated request. If the server returns Access-Control-Allow-Origin: null and Access-Control-Allow-Credentials: true, it is exploitable. Confirm with a PoC: create an HTML file with a sandboxed iframe whose srcdoc fetches the endpoint and postMessages the result back to the parent page.

Bypass One-Liners

<iframe sandbox="allow-scripts" srcdoc="<script>fetch('https://victim.com/api',{credentials:'include'}).then(r=>r.json()).then(d=>parent.postMessage(JSON.stringify(d),'*'))</script>"></iframe>

Developer Fix

Strict Set-based allowlist — no substring, no null

const ALLOWED = new Set([
  'https://app.example.com',
  'https://admin.example.com',
  // Never add 'null' here
]);

function corsMiddleware(req, res, next) {
  const origin = req.headers.origin;
  // Reject null, undefined, or any non-whitelisted origin
  if (typeof origin === 'string' && ALLOWED.has(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Access-Control-Allow-Credentials', 'true');
    res.setHeader('Vary', 'Origin');
  }
  next();
}

Red Flags in Code Review

⚠ origin === 'null' or ALLOWED.includes('null') — null origin trusted
⚠ origin.includes('trusted.com') — substring matching bypassed by crafted hostnames
⚠ Unanchored regex: /trusted\.com/ — must be /^https:\/\/trusted\.com$/
⚠ Falling back to wildcard when origin is not in allowlist: res.setHeader('ACAO', '*')
⚠ Missing Vary: Origin — shared caches may serve wrong ACAO value to other origins
