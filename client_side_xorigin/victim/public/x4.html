Reverse Tabnabbing

The community profile page renders user-submitted links with target="_blank" but without rel="noopener". When a user clicks the link, the opened page gains a reference to the original tab via window.opener and can silently redirect it to a phishing clone.

👤 Community Profile Links

Users can submit their personal website URL to display on their profile. The link is rendered with target="_blank" — no rel="noopener".

WARNING (intentional vulnerability): This link is rendered without rel="noopener". Clicking it will open the URL in a new tab and give that tab access to window.opener, which references this tab.
🔎 How Reverse Tabnabbing Works
Step 1 — Victim clicks the link Victim tab (localhost:3000/x4.html) opens attacker page in a new tab Step 2 — New tab checks window.opener window.opener !== null ← reference to the victim tab The Same-Origin Policy prevents reading the opener's DOM, but allows setting its location. Step 3 — Silent redirect window.opener.location = 'http://localhost:3001/x4_phish.html' Victim switches focus back to the original tab — sees the phishing clone. Step 4 — Credential harvest Phishing page looks identical, asks victim to "re-authenticate".

The fix is trivial: add rel="noopener noreferrer" to every target="_blank" link. Modern browsers also apply this by default for cross-origin links, but you cannot rely on browser version.

📜 Vulnerable HTML — Template Rendering 
<!-- server-rendered template — routes/x4.js -->

<!-- BUG: target="_blank" with no rel="noopener" -->
<a href="{{ userUrl }}"
   target="_blank">           <!-- no rel="noopener" -->
  Visit my site
</a>

<!-- SAFE version -->
<a href="{{ userUrl }}"
   target="_blank"
   rel="noopener noreferrer">  <!-- opener reference nulled -->
  Visit my site
</a>
