[X2] CORS Null Origin Exploit

ATTACKER http://localhost:3001 (this page) TARGET http://localhost:3000 → /api/x2/account

The victim server trusts Origin: null in its CORS allowlist. Browsers send Origin: null from sandboxed iframes (those with the allow-same-origin attribute absent). This page exploits that by injecting a fetch into a sandboxed iframe so the browser sends null as the origin — and the server happily allows it.

Section A — Null Origin via Sandboxed iframe

A sandboxed iframe without allow-same-origin in its sandbox attribute causes the browser to send Origin: null for requests made inside it. If the server trusts null, the response is exposed to the attacker parent page via postMessage.

Section B — Regex / Substring Bypass (Theoretical)

Some servers use a substring check like origin.includes('localhost:3000') instead of an exact match. An attacker can register a domain that contains the trusted string to bypass this check entirely.

Vulnerable server-side check:
if (origin.includes('localhost:3000')) { /* trust it */ }

Bypass: Register a domain like localhost:3000.attacker.io and host your exploit there. The substring check passes — the server trusts the attacker's domain.

How to Test This Manually (curl)

# Send a request with a spoofed Origin header that contains the trusted substring. # The server's regex/includes() check will pass, but the origin is attacker-controlled. curl -s -i \ -H "Origin: http://localhost:3000.attacker.io" \ -H "Cookie: session=<your_session_cookie>" \ http://localhost:3000/api/x2/account # Watch for: # Access-Control-Allow-Origin: http://localhost:3000.attacker.io # If you see your spoofed origin reflected back, the regex is bypassable.
