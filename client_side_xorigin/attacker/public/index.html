Exploit Console — Client-Side & Cross-Origin

This is the attacker origin. From here you launch cross-origin attacks against the victim at http://localhost:3000. Each exploit page demonstrates a distinct vulnerability class and shows the raw exploit code so you can understand exactly what is happening and why it works.

Available Exploits

  • X1
    CORS Origin Reflection
    The victim server reflects any Origin value back in Access-Control-Allow-Origin and pairs it with Access-Control-Allow-Credentials: true — allowing this attacker page to read authenticated API responses cross-origin.
  • X2
    CORS Null Origin
    The victim trusts Origin: null. A sandboxed iframe on this page sends requests with no real origin, bypassing the allowlist. Also demonstrates an origin-regex bypass using a lookalike domain.
  • X3
    postMessage Missing Origin Check
    The victim's postMessage handler responds to any sender without checking event.origin. This page opens the victim in a new window, sends a crafted message, and captures the reply — including the flag.
  • X4
    Tabnabbing (Reverse Tab Hijack)
    When a victim clicks a link that opens this page without rel=noopener, this page gains a reference to the opener tab via window.opener and redirects it to a phishing clone of the victim app.
  • X5
    Cross-Site WebSocket Hijacking (CSWSH)
    WebSocket connections carry cookies automatically, bypassing the Same-Origin Policy read restriction. This page opens a WebSocket to the victim server using the victim's session cookie and reads the flag.
Before you start: Open the victim app at http://localhost:3000 in another tab and log in first. The exploits depend on your browser holding an active session cookie for localhost:3000. Then return here and launch the exploit for each lab.
