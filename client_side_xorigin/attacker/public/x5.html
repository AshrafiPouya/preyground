[X5] Cross-Site WebSocket Hijacking (CSWSH)

ATTACKER http://localhost:3001 (this page) TARGET ws://localhost:3000/ws/x5 (victim WebSocket)

WebSocket connections bypass Same-Origin Policy in a key way: the browser automatically includes cookies from the target domain when opening a WebSocket, even if the page initiating the connection is on a different origin. The server must validate the Origin header during the WebSocket handshake — if it doesn't, any attacker page can impersonate the victim.

Why this works: SOP prevents cross-origin scripts from reading HTTP responses, but it does not prevent opening WebSocket connections. The browser sends the victim's localhost:3000 session cookie automatically because that cookie's domain matches the WebSocket target — just as it would for a same-origin connection. If the server only checks the cookie (not the Origin header), any page on any domain can connect as the victim.

The Fix: Validate the Origin Header

// During the WebSocket upgrade handshake, the server receives an Origin header. // A secure server validates it against an allowlist before accepting the upgrade. // Node.js / ws library example: const wss = new WebSocket.Server({ noServer: true }); server.on('upgrade', (req, socket, head) => { const origin = req.headers.origin; // Reject connections from untrusted origins if (origin !== 'http://localhost:3000') { socket.write('HTTP/1.1 403 Forbidden\r\n\r\n'); socket.destroy(); return; } wss.handleUpgrade(req, socket, head, (ws) => { wss.emit('connection', ws, req); }); });
