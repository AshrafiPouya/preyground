Mutation XSS (mXSS)

The naive sanitizer strips obvious XSS vectors, but the browser's HTML parser mutates certain edge-case markup after sanitization — producing executable code from seemingly safe input.

📜Vulnerable Code — naive sanitizer
// Naive sanitizer: removes script tags and common event handlers
function sanitize(html) {
  return html
    .replace(/<script[\s\S]*?>[\s\S]*?<\/script>/gi, '')
    .replace(/on\w+=["'][^"']*["']/gi, '')
    .replace(/on\w+=\S+/gi, '');
}

// Sanitize, then write into innerHTML → mutation can occur here
const safe = sanitize(userInput);
output.innerHTML = safe;   // browser re-parses → mXSS

Hints
#1Sanitizer ≠ safe render
The sanitizer looks at one representation; the browser's parser may produce a different tree when it re-parses the sanitized output via innerHTML.
#2SVG namespace switching
In SVG context, the parser switches to SVG namespace rules. A <style> inside SVG followed by HTML content can cause the parser to reinterpret the following tags differently.
#3Classic mXSS vector
Try: <svg><style></style><img src=x onerror=captureFlag()></svg>. The sanitizer may not strip the onerror because it's inside SVG context during its parse, but when re-parsed by the browser it fires.
