Authentication Bypass via SQL Injection

The single most-cited SQLi example in interviews. Know it cold.

The 30-Second Answer

"An auth bypass uses SQL injection to make the login query's WHERE clause always true — or to comment out the password check entirely — so you authenticate without valid credentials. The classic example is injecting ' OR '1'='1 into the username or password field, turning the conditional into a tautology."

Common Interview Questions

The original query is: SELECT * FROM users WHERE username = '[INPUT]' AND password = '[PASS]'. If you inject ' OR '1'='1 as the password, the query becomes: ... AND password = '' OR '1'='1'. Since OR '1'='1' is always true, the WHERE clause evaluates to true for every row, and the query returns all users. The application typically takes the first returned row and logs you in as that user.
Injecting admin'-- as the username causes the query to become: SELECT * FROM users WHERE username = 'admin'--' AND password = '...'. The -- is a SQL comment delimiter — everything after it is ignored. The password check is eliminated entirely. As long as an 'admin' row exists, you get back that row and log in as admin, regardless of password.
Targeting the username lets you specify which user to bypass — e.g. admin'-- logs you in specifically as admin. Targeting only the password field with a tautology (e.g. ' OR '1'='1) may log you in as whoever is in the first row returned — often not admin. Precision matters in real assessments.

Explain It to a Developer (the Fix)

Use parameterized queries. The query structure should never change based on user input. Additionally: use constant-time comparison for password checking (to prevent timing attacks), and store passwords with bcrypt/argon2 — though note that hashing protects stored passwords, not a query that's injectable before the hash comparison runs.

# WRONG
query = f"SELECT * FROM users WHERE username='{u}' AND password='{p}'"

# RIGHT
cursor.execute("SELECT * FROM users WHERE username=? AND password=?", (u, p))

Gotchas & Follow-Ups

You can still bypass auth by injecting admin'-- into the username, which comments out the password check entirely. You don't need both fields to be injectable — one is enough to eliminate the other.
The ' OR 1=1-- tautology returns all rows, and the app usually takes the first one. In many schemas, the first user is the admin or a privileged account (lowest ID). Even if not, logging in as any user gives you session access, potentially exposing PII, payment info, and secondary escalation paths.
No. Hashing protects stored passwords from being readable if the database is dumped. But auth bypass happens upstream — the injectable query returns a row (or all rows) before the hash comparison ever occurs. The app sees "got a row back = authenticated." Hashing is irrelevant to this attack.

One-Liners to Remember

admin'--
' OR 1=1--
' OR '1'='1
admin'/*

Red Flags to Avoid Saying

❌ "Bcrypt/hashing stops SQL injection auth bypass" — it doesn't; they solve different problems.
❌ "This only works on old apps" — still found on internal tools, IoT panels, legacy admin portals.
❌ "Just use HTTPS" — transport encryption has no bearing on SQL injection in the application layer.
