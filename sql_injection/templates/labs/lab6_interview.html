Time-Based Blind SQL Injection

Know the delay functions per DBMS cold — this is the first follow-up after boolean blind.

The 30-Second Answer

"Time-based blind SQLi is the last resort — when the response is completely identical regardless of input, you make the database pause execution when a condition is true and infer data from how long the response takes. It's slower and noisier than boolean but works even when there's zero behavioral difference in the output."

Common Interview Questions

When there is literally no observable difference in the HTTP response between true and false conditions — same body, same status, same length. This happens on INSERT/UPDATE/DELETE endpoints that return no data, or apps that return identical "success" pages regardless of internal state. Time-based uses response delay as the only channel.
MySQL: SLEEP(n). PostgreSQL: pg_sleep(n). MSSQL: WAITFOR DELAY '0:0:n'. Oracle: dbms_pipe.receive_message('x',n) or heavy queries. SQLite: no native sleep; use recursive CTEs for CPU-based delay. This is one of the highest-signal questions for detecting DB-awareness — wrong answers here stand out.
Use a threshold significantly above baseline latency (e.g., if baseline is 50ms, use a 3s sleep and threshold of 2s). For noisy environments: repeat each request 3 times and take the median; abort if variance is too high. Sqlmap uses a similar averaging approach. The key is distinguishing a deliberate delay from natural variance.

Explain It to a Developer (the Fix)

Same fix as all SQLi: parameterized queries. Note the important lesson: even an endpoint that returns nothing — a "black box" INSERT that always says "success" — is fully exploitable via timing. "It returns nothing" is not a defense.

One-Liners to Remember

'; SELECT SLEEP(5)-- (MySQL — in SELECT/WHERE context)
'; SELECT pg_sleep(5)-- (PostgreSQL)
'; WAITFOR DELAY '0:0:5'-- (MSSQL)
x' AND IF((SELECT SUBSTR(flag,1,1) FROM secrets)='H',SLEEP(3),0)-- (MySQL WHERE context)
x@x.com' AND sleep(3) AND '1'='1 (SQLite INSERT VALUES context — no -- needed)

Red Flags to Avoid Saying

❌ Using SLEEP() for MSSQL or PostgreSQL — SLEEP is MySQL only. Shows shallow cross-DB knowledge.
❌ Not acknowledging jitter as a problem — production networks have baseline latency that can cause false positives.
❌ Treating time-based as the first choice — it's always the last resort; boolean blind is preferred when any signal exists.
