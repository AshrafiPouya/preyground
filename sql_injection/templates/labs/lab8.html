Second-Order SQL Injection

Registration is safe (parameterized INSERT). But when you view a profile, the stored username is re-used in an unparameterized SELECT. The injection fires in a different query, not where you entered it.

Objective
1. Register with a username that contains a SQL payload (e.g. x' UNION SELECT flag,2,3 FROM secrets--).
2. The payload is stored safely — but it fires when the profile page reads it back into an unparameterized query.
3. The profile query has 3 columns: username, bio, avatar.
Key insight: Sanitizing input at entry time doesn't help if the stored value is later used unsafely.
