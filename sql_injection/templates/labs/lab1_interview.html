Detection & Error-Based SQL Injection

Interview prep guide — revise this the night before your AppSec or penetration testing interview.

The 30-Second Answer

"SQL injection is when untrusted user input is concatenated into a SQL query, letting an attacker change the query's logic. Error-based is the variant where the database's own error messages leak data back to the attacker — they craft a payload that causes an error containing the value they want to extract."

Common Interview Questions

SQL injection occurs when user-supplied input is concatenated directly into a SQL query without proper sanitization or parameterization. The attacker can "break out" of the intended data context and inject their own SQL logic — changing WHERE clauses, appending UNION queries, or triggering errors that leak information. The root cause is always the same: treating user input as SQL code rather than data.
Start with a single quote '. In string context, this breaks the query syntax and causes an error. In numeric context (no quotes around the value), a single quote may not trigger anything — instead try 1 AND 1=1 (should work normally) vs 1 AND 1=2 (should change the result). Also try -- to comment out the rest of the query. Any behavioral change — error, different results, blank page — signals injection.
In string context the input lands inside quotes: WHERE name = '[INPUT]'. You need to close the quote to escape. In numeric context there are no quotes: WHERE id = [INPUT]. You can inject SQL math or functions directly without escaping. This matters because naive "escape all quotes" filters don't protect numeric context at all.
Verbose errors reveal query structure, table names, column names, data types, and sometimes data values directly. With MySQL's extractvalue() or updatexml(), an attacker can craft a payload where the error message literally contains the value they're after (e.g., a password hash or secret). Even without error-based extraction, errors confirm the injection exists and reveal context. Production apps should suppress errors and log them server-side instead.

Explain It to a Developer (the Fix)

Use parameterized queries (also called prepared statements). Instead of building SQL strings with string formatting, pass values as separate parameters — the database driver handles escaping correctly regardless of content.

# WRONG — vulnerable
query = f"SELECT * FROM users WHERE id = {user_id}"

# RIGHT — parameterized
cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))

Additionally: disable verbose error messages in production. Log them server-side. Never show stack traces or SQL syntax errors to users.

Gotchas & Follow-Up Questions

This is why "escape all quotes" is an incomplete defense — it doesn't protect numeric parameters at all. Parameterized queries are the only reliable fix because they separate data from code structurally.
Several reasons: (1) Numeric context doesn't use quotes. (2) Character encoding attacks (e.g., multi-byte sequences) can bypass naïve escaping. (3) Second-order injection — the value is stored safely but later retrieved and concatenated into a new query. (4) Developer mistakes — consistent escaping in every single code path is hard to guarantee. Parameterized queries solve all of these structurally.

Red Flags to Avoid Saying

❌ "You just filter out apostrophes" — implies you think blacklisting quotes fixes SQLi. Interviewers hear this as a junior, blacklist-only mindset.
❌ "SQLi only works on login forms" — SQLi appears in any SQL-backed input: search boxes, numeric IDs, API parameters, HTTP headers, cookies.
❌ "Error-based is the only type of SQLi" — it's one of many. Blind (boolean, time-based), UNION-based, second-order are all distinct.
❌ Claiming a WAF alone fixes the problem — WAFs can be bypassed and are mitigation, not remediation.
