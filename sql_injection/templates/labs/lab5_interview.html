Boolean-Based Blind SQL Injection

"How would you exploit SQLi with no visible output?" — a classic depth-check question.

The 30-Second Answer

"Blind boolean SQLi extracts data even when nothing is reflected in the response — you ask the database true/false questions and read the answer from how the page changes. One character at a time, you test conditions against each position of a string, inferring the value from whether the page behaves as 'true' or 'false'. It's slow but reliable, and it works whenever there's any behavioral difference between true and false."

Common Interview Questions

In-band SQLi (error-based, UNION) returns data directly in the HTTP response. Blind SQLi returns no data — you infer it indirectly. Boolean blind uses page behavior (different content, status code, response length). Time-based blind uses response delay. The injected query logic is the same; only the exfiltration channel differs.
Ask binary questions about each character: "Is the first character of the password greater than 'M'?" If the page responds as 'true', you know it's in the range N–Z. Binary search on ASCII values (~7 questions per character) vs linear search (up to 94 questions per character). For each position, test: AND ASCII(SUBSTR(secret,pos,1)) > midpoint-- and narrow down.
Many things: different page content ("exists"/"not found"), HTTP status code (200 vs 500), response body length (even one character difference), presence or absence of a specific element, redirect vs no redirect. Any reliable binary difference between true and false works as the signal.
Use boolean when there's any observable behavioral difference between true/false — it's faster and less noisy. Use time-based when the response is completely identical regardless of the condition — timing becomes the only signal. Time-based is slower and subject to network jitter. Boolean is always preferred when available.

Explain It to a Developer (the Fix)

"No data shown" is not a defense. The fix is the same: parameterized queries. Even if your app returns zero visible output from injection, an attacker can still extract every byte of your database — it just takes more requests.

One-Liners to Remember

admin' AND 1=1-- (true signal)
admin' AND 1=2-- (false signal — confirm the binary signal works)
admin' AND SUBSTR((SELECT password FROM users LIMIT 1),1,1)='a'--
admin' AND ASCII(SUBSTR((SELECT password FROM users),1,1))>77-- (binary search)

Red Flags to Avoid Saying

❌ Treating boolean and time-based as the same — interviewers want you to distinguish the signal (content vs delay).
❌ "If there's no output, the app is safe from SQLi" — blind injection refutes this directly.
❌ Not mentioning binary search as an optimization — linear A→Z scanning shows you haven't thought about efficiency.
