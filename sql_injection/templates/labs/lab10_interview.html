The Hunt: Methodology & Chaining

"Walk me through how you'd approach an unknown target" — the methodology question that defines senior interviews and practical exams (OSCP-style).

The 30-Second Answer

"Real testing isn't 'apply technique X' — it's mapping the application, identifying every input that reaches SQL, finding the one that behaves anomalously, identifying what constraints exist, and chaining the right techniques to extract data. Methodology beats memorized payloads. The order is always: map → detect → fingerprint → bypass → extract → report."

The Hunt Methodology (in order)

  1. 1. Map — enumerate every input surface: URL params, form fields, JSON body, HTTP headers (User-Agent, Referer, X-Forwarded-For), cookies. Anywhere user data touches SQL.
  2. 2. Detect — probe each input for anomalies: errors, different response lengths, timing changes, behavioral differences. A single quote, a number where text is expected, a doubled apostrophe — any deviation signals injection.
  3. 3. Fingerprint — identify the DBMS via version functions, error messages, and syntax. Never assume MySQL. Confirm before crafting dialect-specific payloads.
  4. 4. Bypass — identify what's blocking you (WAF, input validation, length limits) and select the appropriate evasion. Start with the simplest bypass and escalate.
  5. 5. Extract — enumerate schema, locate the sensitive data, pull it out using the appropriate technique (UNION, boolean blind, time-based) given what the app exposes.
  6. 6. Report — document with a clear repro (URL, method, parameter, payload, response), impact assessment, and remediation recommendation. Findings without reproducible steps aren't findings.

Common Interview Questions

Start with reconnaissance — understand the app's purpose, tech stack (headers, error pages, source), and auth model. Map every input. Passively browse to capture all parameters. Then actively probe each input with detection payloads, watching for anomalies. Prioritize inputs that look user-controlled and SQL-shaped (IDs, search terms, filters). When you find an anomaly, fingerprint the DB, identify constraints, choose the right technique, and extract methodically. Document every step.
Numeric parameters (IDs, counts) are high priority — they're often in numeric SQL context with no quoting. Search fields with visible results are good for UNION. Auth forms are good for bypass. Inputs that have a different response length for different values suggest conditional behavior. HTTP headers and cookies are often overlooked and less tested. Prioritize by: output reflection → observable behavior → numeric context → auth.
Based on what the app exposes: (1) Verbose errors → error-based. (2) Data reflected in response → UNION. (3) Boolean behavioral difference but no data → boolean blind. (4) Identical response every time → time-based. These aren't mutually exclusive — start with the least-intrusive (error detection), escalate to the most powerful (UNION), fall back to blind if needed.
Time-box each input. If after 10-15 minutes of creative probing you see no anomaly, move on and return later. Keep notes on what you tested and what the baseline response is. Use tooling (Burp) to capture and diff responses rather than eyeballing them. Don't assume an input is vulnerable just because it should be — verify every claim. The goal is evidence, not intuition.

Explain It to a Developer (the Fix)

A single missed parameterization anywhere in the codebase can be chained into full database exfiltration. Security must be consistent across every input and every query — not just the obvious ones. Defense-in-depth: parameterized queries (primary), least-privilege DB accounts (limits blast radius), WAF (speed bump), error suppression (removes information), monitoring (detects exploitation).

Technique Decision Tree

Start: found injection point ├── Errors visible? → Error-based (extractvalue/updatexml) ├── Data reflected in output? │ └── Yes → UNION-based │ ├── Count columns (ORDER BY) │ ├── Find reflected column │ └── UNION SELECT flag FROM ... └── No output / no errors ├── Response changes? → Boolean blind (SUBSTR/ASCII) └── Response identical? → Time-based (SLEEP/pg_sleep) Always first: fingerprint the DB engine Always after: enumerate schema before extracting

One-Liners to Remember

map → detect → fingerprint → bypass → extract → report
Any behavioral difference = potential injection signal
Output type determines technique: errors=error-based, data=UNION, behavior=blind, nothing=time
Methodology beats memorized payloads

Red Flags to Avoid Saying

❌ "I'd just run sqlmap" — fine as a tool, fatal as your entire methodology answer. Interviewers want to know you understand the underlying mechanics.
❌ Only knowing one technique — if UNION fails, do you know to fall back to blind? Senior testers adapt.
❌ Not mentioning documentation/reporting — findings without clear repro steps aren't actionable and show lack of professional experience.
❌ "I'd test the login form first" — login bypass is the last place modern apps are vulnerable. Numeric IDs and search endpoints are far more common today.
