Database Fingerprinting & Dialect-Specific Syntax

Always identify the engine before crafting payloads. Assuming MySQL everywhere is a junior mistake.

The 30-Second Answer

"Fingerprinting means identifying which database engine is behind the application — via version functions, syntax quirks, and error messages — so you can use the correct dialect. Payloads that work on MySQL silently fail on PostgreSQL, and vice versa. You always fingerprint before you exploit."

Common Interview Questions

Try version functions: version() works in MySQL and PostgreSQL; @@version works in MySQL and MSSQL; sqlite_version() only works in SQLite. Error message formatting also differs — MySQL errors look different from PostgreSQL errors. Comment syntax: MySQL supports # as a comment; PostgreSQL and MSSQL do not. String concatenation: CONCAT() is MySQL; || is PostgreSQL/Oracle/SQLite; + is MSSQL.
Concatenation: MySQL=CONCAT(a,b), PostgreSQL/Oracle/SQLite=a||b, MSSQL=a+b. Comments: MySQL supports #, -- , and /**/ ; others only -- and /**/. LIMIT/TOP: MySQL/PostgreSQL use LIMIT n; MSSQL uses SELECT TOP n; Oracle uses ROWNUM < n. Schema tables: information_schema (MySQL/PostgreSQL/MSSQL), sqlite_master (SQLite), ALL_TABLES (Oracle). Sleep: SLEEP() (MySQL), pg_sleep() (PostgreSQL), WAITFOR DELAY (MSSQL).

Explain It to a Developer (the Fix)

Fingerprinting is only relevant once injection exists. The fix — parameterized queries — makes DB-specific tricks irrelevant, because the payload can never change the query structure regardless of dialect.

One-Liners to Remember

1 UNION SELECT version(),NULL,NULL-- (MySQL/PostgreSQL)
1 UNION SELECT @@version,NULL,NULL-- (MySQL/MSSQL)
1 UNION SELECT sqlite_version(),NULL,NULL-- (SQLite)
1 UNION SELECT banner,NULL FROM v$version-- (Oracle)

Red Flags to Avoid Saying

❌ Assuming MySQL everywhere without confirming — a senior candidate always fingerprints first.
❌ Not knowing that Oracle uses ALL_TABLES instead of information_schema.
❌ Using # as a comment on PostgreSQL — it doesn't work and reveals that you haven't tested cross-DB.
