Second-Order SQL Injection

A classic senior-level differentiator. Many juniors have never heard of it — knowing it cold distinguishes you.

The 30-Second Answer

"Second-order SQLi is when the payload is stored harmlessly at input — often because that query is parameterized — but later retrieved and used in a different, unparameterized query. The injection fires somewhere other than where you entered it. This defeats 'sanitize on input' strategies because the dangerous query isn't the one that received the payload."

Common Interview Questions

It's a two-phase attack. Phase 1: the attacker submits a SQL payload through a safe channel — usually a registration or profile update form that uses parameterized queries. The payload is stored verbatim in the database. Phase 2: a different feature reads that stored value and inserts it into a new, unparameterized query. The injection fires in Phase 2, not Phase 1.
Because sanitization at input only protects the first query. The stored value — even if "clean" from the first query's perspective — becomes dangerous when concatenated into a second query that was written without thinking about the data's origin. The developer of the second query trusted that the database contains clean data. That trust is misplaced.
Data-flow analysis: trace where user-controlled values are stored, then find every code path that reads those values back and uses them in SQL. Pay attention to: profile pages, admin panels that display user data, email systems that format messages, audit logs. Automated scanners miss it because they probe input and output in the same request. Manual code review or two-request tracing is required.

Explain It to a Developer (the Fix)

Every query that consumes stored data must use parameterized statements — not just the queries that directly receive user input. The rule is: treat data as untrusted at every read, not just first entry. Even if the INSERT was safe, the SELECT that later uses that value must also be parameterized.

# WRONG — second query trusts stored data
username = db.get("SELECT username FROM users WHERE id=?", user_id)
db.execute(f"SELECT * FROM profiles WHERE username='{username}'")

# RIGHT — parameterize every query that uses stored values
db.execute("SELECT * FROM profiles WHERE username=?", (username,))

One-Liners to Remember

Register username: x' UNION SELECT flag,2,3 FROM secrets--
Then trigger the second query (view profile, admin panel, etc.)
The payload fires in the second query, not the registration query

Red Flags to Avoid Saying

❌ "Escaping on input solves SQLi" — second-order is the direct counterexample. The payload bypasses input escaping entirely.
❌ Not being able to describe the two-phase nature — vague answers like "the injection is stored" without explaining when/where it fires.
❌ "Scanners would catch it" — automated scanners almost universally miss second-order because they don't trace data across requests.
