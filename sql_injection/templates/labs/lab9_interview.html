WAF / Filter Bypass

Near-constant in real hunting. "How would you bypass a WAF?" is a common curveball in pentest interviews.

The 30-Second Answer

"WAF bypass exploits the fundamental weakness of blacklist-based filters — they can't anticipate every form a payload can take. You encode, comment, case-vary, or reshape the payload so it bypasses the filter's pattern matching while remaining valid SQL to the database. Blacklists always have gaps; the database's parser is far more flexible than any filter."

Common Interview Questions

Case variation: SeLeCt, UnIoN — if the filter is case-sensitive, uppercase mixing bypasses it. Inline comments: UN/**/ION SEL/**/ECT — SQL parsers ignore comments, most WAFs don't reconstruct the keyword. Double encoding: %55NION (URL encoded U). Nested keywords (for naive regex): UNUNIONION — if the filter strips 'UNION' once, the result is 'UNION'. Combine techniques if one alone fails.
Hex literals: 0x48554e54 instead of 'HUNT'. CHAR() function: CHAR(72,85,78,84). Numeric context: if the target value is numeric, no quotes are needed. Type coercion: some DBs coerce hex to strings automatically. These avoid the need for quotes entirely.
Inline comments as whitespace: SELECT/**/flag/**/FROM/**/secrets. Tab character: %09. Newline: %0a. Many SQL parsers accept any whitespace character where the standard expects a space. Parentheses also work as implicit delimiters in some contexts: UNION(SELECT(flag)FROM(secrets)).
A blacklist defines what's forbidden — anything not on the list passes. It's impossible to enumerate all representations of SQL keywords across encoding schemes, case combinations, and comment styles. An allowlist defines what's permitted — everything else is rejected. For SQL injection specifically, the allowlist is: parameterized queries, full stop. Parameterized queries don't need a WAF because input can never change query structure.

Explain It to a Developer (the Fix)

A WAF is a layer, not a fix. Parameterized queries make WAF bypass irrelevant — even if the bypass works and the payload reaches the database, parameterization ensures it's treated as data, never as SQL code. The WAF is additional defense-in-depth, not a substitute for fixing the root cause.

Red Flags to Avoid Saying

❌ "A WAF makes the app secure" — WAFs are bypassable and are mitigation, not remediation.
❌ Only knowing case variation — senior testers know the full toolkit: encoding, comments, hex, CHAR(), nested keywords.
❌ "Just block all SQL keywords" — this breaks legitimate queries and is still bypassable.
